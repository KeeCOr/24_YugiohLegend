문제 정의

어떤 사람이: 유희왕·매직·개더링 같은 카드 배틀 게임을 좋아하지만 방대한 룰과 긴 진행 시간 문제에 캐주얼하게 즐기기 어려운 사람.

어떤 상황에서: 출퇴근·점심 시간 등 10분 이외의 짧은 여유 시간에 빠르게 한 판을 마칠 수 있는 복잡한 규칙을 몰라도 현재 상황에서 최선의 수를 직관으로 파악하고 싶은 상황.

Pain (핵심 고통): 복잡한 콤보나 전략의 깊이를 즐기려 해도 현재 가능한 선택지가 한눈에 들어오지 않고 즉각적인 결과 피드백이 없어, 짧은 세션에서는 충분한 전략적 만족감이 충족되지 않아 이탈한다.

주 페르소나

박성현 (24세) 대학교 4학년 / 유희왕·하스스톤 경험자

  • 현재 게임 경험: 고등학교 시절 유희왕 카드를 수집·대전했고, 대학 입학 후 하스스톤과 마스터 듀얼을 플레이했다. 현재는 취업 준비로 시간이 부족해 긴 세션의 카드 게임에서 이탈했지만, 대전 카드 게임 특유의 "심리전과 전략 배치" 감각은 여전히 그리워한다.
  • 현재 행동(대안 게임): 유희왕 마스터 듀얼을 가끔 켜지만 한 판이 너무 길고 룰이 복잡해 10분 이내에 접는다. 하스스톤은 P2W 구조에 지쳐 그만뒀다. "빠르게 한 판 대전하고 전략적으로 이기는 맛"을 원하지만 마땅한 대안이 없다.
  • 이 게임을 플레이하는 맥락: 강의 공강 시간 또는 자취방에서 취침 전 10~15분. 규칙을 외우지 않아도 직관적으로 "이 레인에 뭘 배치할지" 결정할 수 있기를 원함. 이기면 "내 전략이 통했다", 지면 "다음엔 다르게 해봐야지" 느낌이 즉시 와야 재시작한다.
  • 특이점: 카드 게임 대전 경험이 있어 로직 파악이 빠르다. 복잡한 콤보보다 "상황 판단 + 배치 결정"의 즉각적인 인과관계를 선호. 친구와 대전하고 싶지만 스케줄 맞추기 어려워 솔로 AI 대전을 선호.

핵심 루프

플레이어가 현재 카드 상황(LP·레인·손패)을 확인
  → 이번 턴 → 몬스터/마법/함정 배치 결정
  → 행동 커밋 → 동시 공개 → 레인별 전투 자동 처리
  → LP 갱신·결과 피드백 확인
  → 보상으로 강화된 덱원으로 다음 턴 준비
  → 다시 "현재 카드 상황 확인"으로 돌아감

패배 이후 메타 루프:

전적 결과 확인
  → 덱 빌더에서 카드 조합 조정
  → 다음 판 시작
  → 다시 핵심 루프

레퍼런스 분석

1. 유희왕 마스터 듀얼 (모바일·PC)

  • 핵심 행동까지 UX 단계 수: 설치 → 계정 생성 → 튜토리얼(약 15분) → 첫 상대 전투 = 6~8단계, 20분 이상
  • 강점: 원작 룰 완벽 재현, 풍부한 카드 풀
  • 단점: 진입 장벽 높음, 한 당 시간이 길어 모바일 세션에 부적합
  • 적용 교훈: 룰 설명보다 먼저 선택 가능한 상황을 보여주기를 시작, 복잡성 분산 위해 레인 1개만 줄여 복잡성을 분산시킴(YugiohLegend의 레인 금화 구조가 이 교훈을 직접 적용).

2. Clash Royale (모바일)

  • 핵심 행동까지 UX 단계 수: 앱 실행 → 튜토리얼 배틀 시작 = 2단계, 90초 이내
  • 강점: 즉시 플레이 가능, 짧은 세션(3분), 전략적 전략
  • 단점: 카드 집결 P2W 구조, 덱 빌딩 깊이 부족
  • 적용 교훈: 첫 플레이까지 단계를 최소화 목표: 앱 실행 후 2단계, 30초 이내 첫 선택). 결과 화면에서 다음 판의 개선 힌트를 즉시 제안.

3. Slay the Spire (PC/모바일)

  • 핵심 행동까지 UX 단계 수: 실행 → 캐릭터 선택 → 첫 전투 = 3단계, 2분 이내
  • 강점: 덱 빌딩 심화성, 극명한 선택지와 전략이 명확
  • 단점: 단일 플레이어, 경쟁 요소 없음
  • 적용 교훈: 카드 효과를 "발동 조건 / 효과 결과" 2줄 구조로 단순화해 보는 서식을 명확화(v0.4.6에 직접 반영).

게임 개요

YugiohLegend는 유희왕을 단순화한 4레인 카드 전투 게임이다. 양쪽 플레이어가 비공개로 행동을 입력하고 동시에 공개하면 서버가 전투를 자동 처리한다. 매 번의 선택은 다음 화면에서의 위험성·보상·성취 방향으로 이어지는 구조를 중심으로 한다.

  • 스택: Phaser 3 클라이언트 + Node.js WebSocket 서버 + Electron 래퍼
  • 화면 기준 해상도: 900 × 1600 (세로형)
  • 현재 버전: 0.4.6

레인 금화

  • 1턴: 기본 레인만 사용
  • 2턴: 금화 + 기본 레인
  • 3턴: 3개의 레인
  • 4턴: 4개의 레인 모두 활용
  • 마지막 레인에서는 몬스터·마법 배치·함정 발동 서버가 처리
  • 클라이언트는 마지막 레인의 자동적인 전투 결과를 활용해 결과를 표시

턴 흐름

1턴(방어):

  1. 기본 레인에서 행동 입력
  2. 몬스터 몬스터, 마법 배치, 함정 발동
  3. 행동 공개
  4. 전투 없이 카드/LP 상태만 갱신함

2~4턴:

  1. 기존 지속 마법 이어받는 것을 갱신
  2. 각쪽 행동 공개
  3. 금화의 레인에서의 행동 적용
  4. 레인별 전투 처리 결과
  5. LP 갱신 및 0 이하가 되면 최종 결과

지속 마법 & 함정

  • 마법은 설치하는 곳까지의 레인에 효과 발동
  • 각 상대 마법·함정의 지속 발동 효과가 쌓이면서 표시
  • LaneState.spell = { card, remainingTurns } 구조로 구현
  • 턴 행동 적용 후 기존 마법의 remainingTurns 감소; 0이 되면 효과 발동 후 삭제
  • 발동 후 해당 레인에 직접 공격하면 마법 무효화, 효과 역전

함정 시스템 (v0.4.1 추가)

  • 조건부 발동 카드(mirror_snare, direct_shield, overextend_collapse)에 trap 타입
  • 함정 선택지에서 뒤집기/공개 가능 여부를 표시하고 나머지 공개됨 face-up spell과 구분
  • AI에서 함정을 뒤집을 경우의 경우 전투 전략을 달리 적용해 정보 인식

카드 효과 가독성 (v0.4.6)

  • 카드 효과 텍스트를 발동 조건 / 효과 결과 2줄 구조로 분리
  • 선택·취소 가능 텍스트에서의 동일 효과 발동 수의 사용 후 카드 선택 화면의 이펙트 표시, 현재 텍스트가 모두 선택 시 배치 제거
  • 비ASCII 캐릭터로 선택된 빌드 분기 피드백을 위해 각 턴 버튼·이벤트 표시 캐릭터를 ASCII 기반으로 정리

화면 구조

  • 게임 화면: 각 측 카드 현황·전투 결과 카드·손패 (세로형 스택)
  • 덱 빌더: 카드 선택창 3턴 + 현재 덱 구성 목록

설계원칙과 UX 지침

  • 주요 버튼 44px 이상, 화면당 CTA 강조색은 1개
  • 버튼·선택지 매 화면에 5개 이하 (Hick's Law)
  • 결과 피드백 400ms 이내 표시 (Doherty Threshold)
  • 로딩·오류 상태·비움·장문 데이터 빈 상태 스켈레톤 UI도 준비
  • HUD 동일 레이어 요소 최소 겹침 원칙; 겹침 필요 시 별도의 레이어도 depth/z-order 사용

개선된 UX / UI

  • 레인 현재 상태 정보 분산 방지: 현재 자동으로 전투 결과를 표시 → "N개의 마지막 금화" 이어받는 것에서의 화살표를 레인 기본에서 방향으로 표시하고, 마지막 레인에 남기게 되면 "2판에 걸쳐 레인이 달라진다" 일괄 표시. 따라서 플레이어가 레인 구조를 3판 이내에 직관으로 파악하도록 원래대로 고려.
  • 공격 버튼 피드백 배치: 행동 입력 후 공격 카드마다 "현재 발동 가능한 행동 발동" 안내 목록(레인별 배치 카드 선택 최소 1개 이상의 공격 버튼을 선택마다 표시. 입력 수를 공격 카드에 반영하도록 → "더 이상하고 지는 결과" 분기 감소.
  • 결과 화면 이후 개선 제안 배치: 결과 화면 이후에서 "다음 판에서는 이것도 가능" 카드 1~2개를 앞의 판에서 이렇게 이용할 것을 제안. Clash Royale의 "다음 개선 힌트를 즉시 제안" 교훈을 직접 적용.
  • 손패 카드 구분 가독성: 몬스터·마법·함정 카드 아트가 동일 색·동일 스타일이라 구분이 힘든 것의 이펙트에서 카드 구성 및 능력의 몬스터(붉은색)/마법(초록색)/함정(주황색)으로 색상 적용하고, 손패 영역에서의 상단 라벨 안내 선택 최소를 표시.

중기 기능들의 가능

  • 카드 풀 마법 카드 선택 (자체적인 로컬): 카드 풀의 기본 20장 + 마법 자체적인 15장, 구조로 분리해서 이 분야 1,000~2,000판에 이름. 함정·마법의 선택 기준은 "가능한지의 여부와 공격 상대 타입 몬스터 선택 기준은 "공격하는지의 기준"으로 플레이어의 탐색과 피드백 경험도 도움. P2W 피드백을 위해 최고의 카드는 기본 카드에 구별 나중 방향으로 남겨둠.
  • AI 전투 로컬 대기 (설정됨 추가): 매일 업데이트되는 30종류 AI 로컬(스타일·난이도) 선택 후 전용 카드 뒷면 앞의 금화. 약 500~1,000번의 업데이트로 고급 이후에서 정보의 기능에서 표시를 알림. 어느 기에서도 1~10종류의 최대 사용 전략.
  • 덱 슬롯 마법 카드: 기본 3개의 덱 슬롯에 5개까지 최대, 이후 슬롯에 300번 확장. 덱 빌딩 깊이를 즐기는 덱별 이후에 가능한 상황.

전기 개발 마법 카드 확장

  • 초기(1~2개월): 카드 풀 20개~40장 확장(레인 방어를 위한 공격 카드 10개 + 공격 상대 타입 몬스터 10개 확장), AI 스타일 3종류 구현, 결과 화면 경험 힌트 목록 확장
  • 이후(3~6개월): 실시간 1v1 매칭 시스템(WebSocket 기반 또는 구조 활용), 랭킹 매칭 + 역할 리더보드, 마법 자체적인 3종의 공격이상의 방어 이후, 솔직한 시간으로 이탈은 결과 덱 빌딩 흐름 확인 어려워
  • 장기(6개월+): 2v2 협력 방향 및 팀(모두의 각 2인이 모두 2개의 레인에서 해당), 일정 순서 카드 풀 배치 이후의 덱 "이번 기본 마법 카드 사용 분기"), 카드 아트 자체적인 아트 신규 구현해 이후 시작 카드 뒷면 충성)

잔여 기술적인 개선

  • WebSocket 재연결 추가 방지 구조: 모바일 환경에서 네트워크 단절 시의 게임 상태가 달라지는 로드가 서버가 sessionId 기반으로 30분동안 게임 상태를 유지하고, 재연결에 따른 현재 턴 상태를 기존 전달하여 추가 데이터 확장. 실시간 매칭 활성 확인 때의 구현.
  • Phaser setDepth 설정 가능성 확인 동작: HUD 레이어 겹침 교차가 실제 확인에서 선택 하위의 이후 마지막 레이아웃의 다수 게임·클라이언트 depth 값을 최적화하여 이상을 확인하는 매우 다수 확장. depth 계획(동일 값 + 부작용 조합) 정기적인 확인 후 빌드 확인.
  • 카드 데이터 레버리지 삭제: 현재 카드 효과 텍스트가 클라이언트 수로 분산된 후 cards.json 단일 파일에 내림하고, 빌드 이후의 확인 테스트를 확장해 카드 추가/정정 이후에서 각 이후 추가 없이 반영 가능하도록 구조. 카드 풀 마법 카드 마법 이상 2번 이상 확인 예상.
  • Electron 내장 HTTP 서버 고정 포트 피드백: 고정 포트 사용 후 동일 PC에서 다수의 데스크톱 실행의 제안 → 추가 시작 후 사용 가능한 포트를 최적으로 이를 개선하는 구조 확장(포트 기준: 14000~14099).
