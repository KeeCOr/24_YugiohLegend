# YugiohLegend — 게임 설계 문서

**날짜:** 2026-05-14
**레포:** https://github.com/KeeCOr/24_YugiohLegend.git
**스택:** Phaser 3 (클라이언트) + Node.js WebSocket (서버)

---

## 1. 게임 개요

유희왕을 대폭 단순화한 3턴 카드 대전 게임.
양쪽 플레이어가 비공개로 행동을 입력한 뒤 동시 공개하여 전투를 해결한다.
웹으로 시작해 스팀(Electron), 모바일(Capacitor)로 확장한다.

---

## 2. 게임 규칙

### 기본 수치
| 항목 | 값 |
|------|-----|
| 시작 LP | 4000 |
| LP 0 즉시 패배 | O |
| 총 턴 수 | 3턴 + 파이널 배틀 페이즈 |
| 시작 핸드 | 4장 |
| 매턴 드로우 | 1장 |
| 매턴 소환 | 1장 (일반 소환) |
| 마법/함정 사용 | 무제한 |
| 덱 크기 | 8~12장 |
| 같은 카드 최대 | 2장 |

### 필드 구조
- 각 플레이어 앞에 3개의 레인(몬스터존)
- 레인 번호는 1~3, 양쪽 동일한 번호끼리 정면으로 마주봄
- 수비 표시 없음, 공격 표시만 존재

### 전투 규칙
- 몬스터를 소환하면 매턴 자동으로 정면 레인의 상대 몬스터를 공격
- 정면 레인이 비어있으면 상대 LP에 다이렉트 어택
- 양쪽 공격은 동시에 해결됨 (선공/후공 없음)
- 두 몬스터가 서로 공격 시: 공격력이 높은 쪽이 낮은 쪽을 파괴, 차이만큼 상대 LP 감소
  공격력이 같으면 양쪽 모두 파괴, LP 변화 없음

### 카드 종류
| 종류 | 설명 |
|------|------|
| 몬스터 | 공격력 보유, 레인에 소환, 매턴 자동 공격 |
| 마법 | 즉발 효과만 (지속 마법, 속공 마법 없음) |
| 함정 | 세트 후 조건 달성 시 자동 발동 (수동 발동 없음) |

### 턴 흐름
```
[매 턴]
1. 양쪽 드로우 (1장)
2. 양쪽 행동 입력 (비공개)
   - 몬스터 소환 (레인 선택, 1회)
   - 마법 사용 (무제한)
   - 함정 세트 (무제한)
3. 제출 버튼 클릭
4. 서버: 양쪽 모두 제출 확인
5. 동시 공개 애니메이션
6. 전투 자동 해결 (동시)
7. LP 갱신, 0 이하면 즉시 패배 판정

[3턴 종료 후 — 파이널 배틀 페이즈]
8. 필드의 모든 몬스터가 한 번씩 공격 수행
9. LP 비교 → 높은 쪽 승리 (동점이면 무승부)
```

## 3. 아키텍처

### 클라이언트 (Phaser 3)
```
src/
  scenes/
    BootScene.ts        - 에셋 로딩
    MenuScene.ts        - 메인 메뉴
    DeckBuilderScene.ts - 덱 빌더
    GameScene.ts        - 메인 게임 필드
    ResultScene.ts      - 승패 결과
  components/
    Field.ts            - 필드/레인 렌더링
    CardSprite.ts       - 카드 오브젝트
    HandArea.ts         - 핸드 UI
    LPDisplay.ts        - LP 표시
  network/
    SocketManager.ts    - WebSocket 통신
  ai/
    AIClient.ts         - 싱글 모드 AI 로컬 인터페이스
  data/
    cards.json          - 카드 데이터
    CardTypes.ts        - 타입 정의
```

### 서버 (Node.js + ws)
```
server/
  index.ts             - 서버 진입점
  RoomManager.ts       - 방 생성/관리
  GameRoom.ts          - 게임 상태 및 턴 처리
  BattleResolver.ts    - 전투 해결 로직
  AIEngine.ts          - AI 행동 결정
  types.ts             - 공유 타입
```

### 통신 프로토콜 (WebSocket 메시지)
| 방향 | 이벤트 | 내용 |
|------|--------|------|
| C→S | `join_room` | 방 참가 (싱글/멀티) |
| S→C | `game_start` | 초기 핸드, 선공 정보 |
| C→S | `submit_action` | 턴 행동 제출 |
| S→C | `reveal` | 양쪽 행동 공개 |
| S→C | `battle_result` | 전투 결과, LP 변화 |
| S→C | `game_over` | 승패 결과 |

## 4. 카드 데이터 구조

```typescript
type CardType = 'monster' | 'spell' | 'trap';

interface Card {
  id: string;
  type: CardType;
  name: string;
  atk?: number;          // 몬스터만
  effect?: string;       // 효과 ID (null이면 무효과)
  trapCondition?: string; // 함정 발동 조건 ID
}
```

**공격력 범위:** 500 ~ 3000 (500 단위)

## 5. AI 설계

| 난이도 | 전략 |
|--------|------|
| 랜덤 | 유효한 행동 중 무작위 선택 |
| 그리디 | 공격력이 가장 높은 몬스터를 가장 위험한 레인에 배치 |

- AI는 서버 사이드에서 동작
- 클라이언트 입장에서는 온라인 상대와 동일한 흐름
- AI 딜레이: 0.5~1.5초 (자연스러운 느낌)

---

## 6. 플랫폼 확장 계획

| 단계 | 플랫폼 | 방법 |
|------|--------|------|
| 1 | 웹 | Phaser + WebSocket 서버 |
| 2 | 스팀 | Electron으로 클라이언트 래핑 |
| 3 | 모바일 | Capacitor로 클라이언트 래핑 |

서버는 모든 단계에서 동일하게 유지.

---

## 7. 승패 조건 요약

1. 상대 LP가 0 이하 → 즉시 승리
2. 3턴 + 파이널 배틀 후 LP 비교 → 높은 쪽 승리
3. LP 동일 → 무승부
